# 项目审阅 — 2026-04-25

`user-management` 全量代码走查，按严重度归类。每条给出：位置、现象、影响、建议修法。

## 1. 严重 / 行为缺陷

### 1.1 代理配置无开关、URL 硬编码（用户已提）
- **位置**：`templates/proxy.sh`、`lib/ops/create_user.sh`、`lib/interactive/cmd_add_user.sh`、`bin/create-managed-user.sh`
- **现象**：
  - `proxy.sh` 把 `http://127.0.0.1:7890` 写死，新建用户 `~/.bashrc` 必然写入此段。
  - 创建用户时只有「是否部署 scripts」一个开关，`scripts` 与「写入 proxy 段」绑定，无法只复制脚本不写代理。
  - 目标机如果没有本地 7890 代理，新用户每次登录 shell 都会因环境变量设置错误代理，访问外网受影响。
- **建议**：
  - 拆分两个开关：是否部署 `~/scripts`（已有）；是否在 `.bashrc` 写入 proxy 段（新增）。
  - 默认值 `UM_CONFIGURE_PROXY_DEFAULT` 默认 `false`，避免误配置。
  - 代理地址抽到环境变量 `UM_PROXY_URL`，方便自定义。

### 1.2 proxy 模板的注释紧挨赋值（用户已提）
- **位置**：`templates/proxy.sh`、写入 `.bashrc` 时 `BEGIN/END` 锚点注释紧贴 `export`。
- **现象**：当模板加注释（如「proxy 地址，可改」）时，若注释紧贴 `export http_proxy=...`，可读性差；写入 `.bashrc` 时锚点注释也直接挨着 `export`。
- **建议**：注释与赋值之间留一空行；同时 `BEGIN/END` 锚点行与 `proxy.sh` 内容之间也加空行。

### 1.3 JSON 由 heredoc 拼字符串生成，未转义
- **位置**：`lib/ops/create_user.sh:73-87`
- **现象**：`authorized_keys`、`home_dir` 等直接插入到 JSON 字符串。`authorized_keys` 含 `"`、`\` 时 JSON 直接坏掉；后续 `_load_user_data` 也读不出。
- **建议**：交给 `python3 -c 'import json; json.dump(...)'`（项目已强依赖 python3），统一替代手写 JSON。

### 1.4 字段解析用 grep+sed 易脆
- **位置**：`lib/user_json_parse.sh`、`bin/list-managed-users.sh`、`bin/show-managed-user.sh` 等
- **现象**：
  - `login_ips` 仅取首元素，且假定 IP 不含 `:`（不支持 IPv6）。
  - 多行格式化 JSON 直接挂掉。
  - 「最后一个字段无逗号」场景下 `[^,]*` 会把 `}` 一并捕获。
- **建议**：同上，用 python3 单点解析。或加最小 `jq` 检测（不存在则 fallback）。

### 1.5 `bin/enable-user-sudo.sh` / `disable-user-sudo.sh` 不刷新 JSON
- **位置**：`bin/enable-user-sudo.sh:40-43`、`bin/disable-user-sudo.sh:40-41`
- **现象**：交互菜单里 `_enable_sudo` / `_disable_sudo` 改完系统会调用 `_merge_json_sudo_from_system` 同步 JSON；独立 `bin/` 脚本不同步，导致 JSON 失真。
- **建议**：在两个 `bin/` 脚本里也调用 `_merge_json_sudo_from_system "$json_file" "$username" refresh`。

### 1.6 `bin/create-managed-user.sh` 提示与 HOST_IP 顺序矛盾
- **位置**：`bin/create-managed-user.sh:70-80`
- **现象**：先 `echo "选择登录IP:"`，再判断 `HOST_IP`，导致 HOST_IP 已设也照样打印「选择登录IP:」标题。
- **建议**：把 `echo "选择登录IP:"` 移到 `else` 分支内。

### 1.7 `cmd_add` `select` 错误处理逻辑不通
- **位置**：`lib/interactive/cmd_add_user.sh:80-85`
- **现象**：`select ... done 2>/dev/null || selected_ip="127.0.0.1"`。`select` 成功不会触发 `||`，且通常不返回非零；fallback 不会生效。
- **建议**：手动判断 `available_ips` 是否为空，空则直接 `selected_ip="127.0.0.1"`。

### 1.8 密码明文回显
- **位置**：`cmd_add` line 24、`bin/create-managed-user.sh:27`
- **现象**：`read -p "初始密码: " password` 输入时屏幕显示。
- **建议**：改 `read -srp "初始密码: " password; echo`；或加确认回读。

### 1.9 `setup-dev-env.sh` 把 MiniMax key 误配成 OpenAI 官方端点
- **位置**：`templates/setup-dev-env.sh:130-137`
- **现象**：
  ```
  export OPENAI_API_KEY="$MINIMAX_COM_KEY"
  export OPENAI_BASE_URL="https://api.openai.com/v1"
  ```
  用 MiniMax key 调 OpenAI 官方域名，必然 401。
- **建议**：要么删掉这段（让用户自己设置），要么用正确端点（如 `https://api.minimax.chat/v1`）。

### 1.10 `setup-dev-env.sh` `~/.claude` 整体跳过
- **位置**：`templates/setup-dev-env.sh:140-170`
- **现象**：检测到 `~/.claude` 存在就直接跳过 `everything-claude-code` 的所有步骤，包括局部更新 agents/rules/...；老用户无法增量同步。
- **建议**：对每个子目录单独判定，或拆出独立子命令。

### 1.11 `proxy.sh` 与 `unset_proxy.sh` 大小写不对称
- **位置**：`templates/proxy.sh`、`templates/unset_proxy.sh`
- **现象**：`setup-dev-env.sh` 用 `HTTP_PROXY/HTTPS_PROXY/ALL_PROXY` 三个大写名；`proxy.sh` 只 export 小写两个；`unset_proxy.sh` 也只 unset 小写。
- **建议**：模板里同时 export 大小写四件套（含 `no_proxy/NO_PROXY`），`unset` 也对称。

## 2. 中等 / 鲁棒性

### 2.1 `deluser` 仅 Debian 系
- **位置**：`lib/interactive/cmd_user_sync_and_sudo.sh:67,101,107`、`bin/disable-user-sudo.sh:41`
- **现象**：CentOS/RHEL 没有 `deluser`，应用 `gpasswd -d "$username" sudo` 或 `wheel`。
- **建议**：包一层 `_remove_from_group`，按可用命令选择。

### 2.2 `useradd` 不指定 `-U`
- **位置**：`lib/ops/create_user.sh:33`
- **现象**：默认在多数发行版会建同名 group；显式 `-U` 更稳。
- **建议**：补 `-U`。

### 2.3 `HOST_NAME` 与 shell 内置 `HOSTNAME` 同名
- **位置**：`lib/config.sh:21`、README 表格
- **现象**：bash 自带 `HOSTNAME`；用户在 `.env` 写 `HOSTNAME=...` 看似覆盖，实际行为受 bash 影响，命名冲突。
- **建议**：改用 `UM_HOST_NAME`，README 同步。

### 2.4 `key_type_inferred` 写入未归一为 bool
- **位置**：`lib/ops/create_user.sh:73-87`
- **现象**：`$key_type_inferred` 直接拼到 JSON。如果上游传 `True/yes/1`，JSON 不合法。
- **建议**：函数入口先归一为 `true/false`。

### 2.5 sed 锚点未引用 `UM_PROXY_BEGIN/END`
- **位置**：`lib/interactive/cmd_user_sync_and_sudo.sh:32`、`bin/reinit-user-environment.sh:38`
- **现象**：sed 字面量与 `lib/config.sh` 中的锚点变量是两套独立写法，未引用变量。
- **影响**：日后改锚点要改三四处，容易漂移。
- **建议**：引用 `$UM_PROXY_BEGIN/$UM_PROXY_END`（注意转义 `(` `)`）。

### 2.6 `_load_user_data` 多元素 `login_ips` 仅取首条
- **位置**：`lib/user_json_parse.sh:30-31`
- **现象**：JSON 字段是数组；其他元素被忽略，UI 只展示一条。
- **建议**：要么文档化，要么循环展示全部。

### 2.7 集成测试覆盖薄
- **位置**：`tests/integration/test_user_lifecycle.sh`
- **现象**：仅创建+删除最小路径，不测 sudo 启用/禁用、proxy 段写入、track 流程。
- **建议**：补 `_enable_sudo`/`_disable_sudo`/`_track_user` 用例。

### 2.8 并发写 JSON 无锁
- **位置**：`lib/ops/create_user.sh`、`json_user_state.sh`
- **现象**：两个并发调用同一用户 JSON 会丢更新。
- **建议**：`flock` 包一层。

## 3. 轻微 / 风格

### 3.1 `bin/` 脚本与交互菜单逻辑重复
- 创建/删除/sudo 启停/同步在 `bin/` 与 `lib/interactive/` 各写一份。
- **建议**：`bin/` 脚本只解析参数后 `source lib/...` 调用统一函数。已部分这样做（create/delete），其余照样收敛。

### 3.2 `cmd_add` 大量 `[[ "$x" == $'\e' ]]` 判断 ESC，分散
- **建议**：抽 `_read_or_back` 工具函数。

### 3.3 README 字段名漂移
- README 「环境变量 .env」表把变量名写为 `HOSTNAME`，与 2.3 一致。
- 改名后 README 同步。

### 3.4 `bin/show-managed-user.sh` SSH config 片段含中文注释
- 复制到本地 `~/.ssh/config` 时虽然 SSH 会忽略 `#`，但跨机粘贴时编码异常会引发意外。

### 第三次（可插拔预装模块 + 模块管理界面 + 锚点块统一）
- **lib/anchors.sh**：通用 BEGIN/END 注释锚点读写（`_um_anchor_write/strip/present`）。所有由本工具写入用户文件的内容统一用 `# BEGIN user_management <name>` ... `# END user_management <name>` 包裹，便于 remove 时只清理自己写入的部分。
- **lib/install_steps.sh**：预装步骤注册表（`UM_STEPS`、`_um_step_call`、`_um_steps_load`）。
- **lib/install_steps/<key>.sh**（每个预装项一个文件）：
  - `scripts_dir` —— 部署 `~/scripts`
  - `proxy_bashrc` —— `.bashrc` proxy 段（含旧锚点迁移）
  - `authorized_keys` —— `~/.ssh/authorized_keys` 锚点段（不再覆盖整文件，保留用户自维护的其他公钥）
  - `sudoers` —— `/etc/sudoers.d/<user>`
  - `docker_group` —— usermod -aG docker
- **lib/install_steps/_template.sh.example**：新增模块的拷贝模板，含 npm/pipx/apt/bashrc/主目录 文件五种范式注释。
- **lib/ops/create_user.sh**：内联 sudoers/docker/authorized_keys/scripts/proxy 全部改为 `_um_step_call <key> apply`，签名不变。
- **lib/proxy_block.sh**：保留为薄壳调 `proxy_bashrc` step（向后兼容）。
- **「修改」菜单加 4) 重新配置预装项**（`_reconfigure_user`）：遍历 `UM_STEPS`，逐项显示当前状态、询问 apply/remove/keep；与创建时的 step 列表一一对应。
- **主菜单加 4) 预装模块管理**（`_modules_menu`）：列模块 + 描述 + 默认值；选模块后可探测/安装/卸载、跨用户状态总览；含「如何添加新模块」帮助。
- 模板 `proxy_bashrc` 保留对旧锚点 `# BEGIN user_management proxy (templates/proxy.sh)` 的 strip，保证旧用户首次重新配置不会留旧块残骸。
- **集成测试** 跟进新加载顺序。
